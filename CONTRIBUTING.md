# 贡献指南

感谢你考虑贡献 `productize-your-skills`。

## 行为准则

- 不提交包含真实客户、订单、内部数据的案例
- 不提交虚假的销量、收入或转化率数据
- 不在示例中包含可识别个人身份的信息
- 不提交未授权转载的内容

## 事实纪律（必读）

本项目 2026-07-14 的 v0.1.0 → v0.1.6 期间，README 出现了 5 次"以推断代替事实"的错误（详见 [`docs/retrospectives/2026-07-14-readme-facts.md`](retrospectives/2026-07-14-readme-facts.md)）。

**所有贡献者必须遵守的事实纪律**：

- **R1**：不知道就明说"我不知道"，不准用"以官方文档为准"等词
- **R2**：写具体路径 / 命令 / API 之前，必须 `curl` 验证
- **R3**：推断和事实分开标注（✅ 事实 / ⚠️ 推断待验证）
- **R4**：用户指出错误时立即改，不解释不辩护
- **R5**：调研一次性做完，不分版本挤牙膏
- **R7**：每条事实配可验证的来源

**PR 会被拒的情形**：

- 包含未验证的路径或 API
- 包含 "通常""应该""以官方文档为准" 等含糊词
- 包含未经 `curl` 验证的链接
- 把推断包装成事实却不标注

## 提交内容

### 1. 报告 Bug

请在 Issue 中提供：

- Codex 版本、操作系统
- 复现步骤和实际行为
- 期望行为和实际行为
- 截图或终端输出（注意脱敏）

### 2. 提出改进

改进前请先开 Issue 讨论，避免重复劳动。

### 3. 贡献案例

案例是这个 Skill 最有价值的资产。

**什么是一个好案例：**

- 来源清晰：你自己设计的，或者你拆解过公开发布的产品
- 字段完整：能填满 `references/case-template.md` 16 个字段
- 有可迁移结构：其他人可以在不同背景上借鉴
- 诚实标注不可照搬的条件：避免误导

**什么不是一个好案例：**

- 标题党式“我靠 XX 一年赚了 100 万”
- 没有具体客户、问题、交付、定价的纯方法论
- 包含来源未授权的内容

**提交流程：**

1. 在 `references/cases/<your-id>.json` 准备文件
2. 运行 `python3 scripts/add_case.py references/cases/<your-id>.json`
3. 运行 `python3 scripts/validate_case.py` 确认无错误
4. 在 PR 中说明：
   - 案例来源
   - 适用范围
   - 不可照搬条件

### 4. 修改代码

修改 `scripts/` 任何文件后：

```bash
python3 -m unittest discover -s tests -v
python3 scripts/validate_case.py
```

- 新增功能必须有对应的单元测试
- 新增对话规则必须有对应的前向测试用例

## PR 流程

1. Fork → 新建分支 → 提交
2. 标题以 `feat:` `fix:` `docs:` `test:` `refactor:` 开头
3. PR 描述中说明：
   - 改了什么
   - 为什么这么改
   - 是否更新了文档和测试
4. 等待 review：典型 SLA 7 天

## 发布

维护者用语义化版本：

- 重大调整 → 主版本
- 新增功能 / 案例 → 次版本
- Bug 修复 / 文案 → 修订号

## 联系方式

请用 GitHub Issue 而非邮件。
