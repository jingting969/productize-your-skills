# 能力产品化

从证据出发，把能力转成可测试产品。不要把兴趣、自我评价或案例类比冒充市场需求。

## 选择模式

先判断用户意图：

- 想发现或设计自己的产品：使用“能力产品化模式”。
- 提供一个成熟产品，希望拆解、保存或加入案例库：使用“案例学习模式”。
- 同时需要两者：先学习案例，再回到能力产品化；案例只能提供类比。

## 能力产品化模式

1. 读取 `references/framework.md`。
2. 需要比较证据或候选方向时，读取 `references/evidence-and-scoring.md`。
3. 需要选择产品形态时，读取 `references/product-patterns.md`。
4. 每轮只问一个最关键的问题。先盘点知识、技能、经历、资源、兴趣和他人求助，再追问具体结果、反馈和付费。
5. 最多提出三个候选方向。对每个方向分开列出事实、推断和待验证假设。
6. 用户确认方向后，设计客户、购买情境、问题、承诺、交付、周期、客户配合、包含项、不包含项和验收标准。
7. 设计试卖价、标准价和升级价，说明价值锚点和涨价条件。未经付费验证的价格必须标注为假设。
8. 输出首批客户画像、名单来源、筛选方法、私聊邀约、公开招募、访谈问题、试卖流程和验证指标。
9. 读取 `references/output-template.md`，根据证据输出正式产品、试卖产品或探索实验。

若用户已有成熟方向，跳过广泛盘点，直接审查证据、需求和交付结构。信息不足时不要补全，改为最小可测试产品和验证任务。

## 案例学习模式

1. 确认资料来源，以及是否允许保存和在后续诊断中引用。
2. 读取 `references/case-template.md`，按统一字段拆解案例。
3. 明确可迁移结构和不可照搬的背景条件。
4. 资料缺失时填写“资料未提供”，降低完整度和可信度，不要编造。
5. 用户要求落盘时，将结构化字段保存为 JSON，并运行：

```bash
python3 scripts/add_case.py /path/to/case.json
python3 scripts/validate_case.py
```

6. 后续产品化需要案例时，先读取 `references/case-index.md`，再只读取最相关的少量 `references/cases/*.md`。
7. 引用案例时说明借鉴的结构；用户自身证据始终优先于案例。

## 必守边界

- 不保证成交、收入或特定业务结果。
- 不在没有证据时断言需求、支付意愿或合理价格。
- 不让结果承诺超过证据等级。
- 不把目标客户写成所有人。
- 不在试卖前建议重资产制作。
- 不以案例替代需求访谈和付费验证。
- 涉及法律、税务、医疗或投资时提醒用户咨询相应专业人士。
