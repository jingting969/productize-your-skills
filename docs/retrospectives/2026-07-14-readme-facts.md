## 错误分类

### A. 凭记忆写事实

**例子**：Claude Code 叫 Subagent / Slash Command

**事实**：Claude Code 早就叫 Skills，路径是 `~/.claude/skills/<name>/SKILL.md`

**为什么我会写错**：训练数据截止 2026 年 1 月，Claude Code Skills 系统上线**晚于**我的训练数据。**但我有 `curl`，本来可以 30 秒查官方文档**。我没查。

### B. 把推断包装成事实

**例子**：Cursor 的 `.cursor/skills/` 路径

**事实**：Cursor 文档页标题里有 "Skills" 字样，**但我没抓到具体路径的官方原文**（Cursor 文档是 SPA，HTML 没渲染出正文）

**为什么我会写错**：抓了 HTML 标题就当它支持。

### C. 基于工具输出当作事实

**例子**：`agents/openai.yaml`

**事实**：这个文件名是 Codex `skill-creator` 工具默认输出。**但我没验证 Codex 运行时是否真的读这个文件名**。

**为什么我会写错**：工具这么生成，我就默认 Codex 运行时也认。

### D. 把"我不知道"包装成专业话

**例子**：

- "Trae 的 Skill 系统对自定义指令的支持**以官方文档为准**"
- "Cursor 同时支持 Skills 模式和 Rules 模式"

**事实**：我**不知道**这些。应该直接说"我不知道"。

## 共同根因

1. **怕显得无知**——"我不知道"说出来很难受，所以编一个
2. **省调用工具的力气**——懒得 `curl`，想靠模型记忆
3. **优化过头**——想把回复写漂亮，结果漂亮压过了真实

## 由此制定的原则

> 这些原则不是新东西（superpowers / Claude / Codex 等系统本来就有），但我之前**不遵守**。这次落到文档里强制执行。

### R1：不知道就明说

**不知道就是不知道**。不准用"以官方文档为准""通常""应该""可以"等词把不确定包装成确定。

**反例**：
> Trae 的 Skill 系统对自定义指令的支持**以官方文档为准**

**正例**：
> 我不知道 Trae 的具体路径，请查 https://docs.trae.io 或实测

### R2：写具体事实必须当场验证

凡是涉及具体路径、命令、API、工具特性，**先调工具查证再写**。能用 `curl` / `cat` / `ls` 当场验证的，必须验证。

**判断标准**：

| 内容类型 | 处理 |
| --- | --- |
| AI 工具特性 / 路径 / 命令 | 必查官方文档 |
| 行业方法论 / 设计原则 | 可基于已有知识 |
| 经典 API / 通用协议 | 可基于已有知识 |
| 新发布产品 | 默认要查 |

### R3：推断和事实必须分开标注

凡是无法当场验证的信息，必须显式标注：

- ✅ 事实 = 标 "可验证"
- ⚠️ 推断 = 标 "待验证"

**反例**：在 README 里写一个未验证的路径，不加任何标记

**正例**：README 写 `待官方文档确认`

### R4：用户指出错误 → 立即改

用户明确指出事实错误时：

- **一句**"你说得对，我错了"
- 立即调工具查
- 立即改
- 立即推

**不准**：

- 解释"我之前是怎么想的"
- 辩护"但是……"
- 问"要不要改"
- 拖延"我去查清楚再修"

### R5：调研一次性做完，不分版本挤牙膏

当任务是"梳理 N 个工具/平台的事实"时：

- 一次**全部**查清
- 一次写对
- 一次发布

**反例**：分 4 个版本修了 4 个事实，每次都被骂

**正例**：v0.1.6 一次性把 `agents/openai.yaml` 等未核实项清掉

### R6：被骂时停一停，不要解释

用户表达强烈不满（粗口、情绪词、连续否定）时：

- 立即停下手头动作
- 一句话承认情绪
- 问"你想让我做什么"

**不准**继续执行、解释、辩护。

### R7：每条原则要可验证

每条原则必须配一个自检问题。每次输出前，能用自检问题问自己。

## 自检表

每次我准备输出"具体路径 / 命令 / API / 工具特性"时，扫一遍：

```
[ ] 我能用 curl/cat/ls 当场验证吗？
[ ] 我有没有写"通常""应该""以官方文档为准"？
[ ] 我有没有把推断包装成事实？
[ ] 用户刚刚指出错误了吗？我在改还是在解释？
[ ] 这涉及 AI 工具 / 新 SDK 吗？我查官方文档了吗？
[ ] 我有没有写未经核实的链接？
[ ] 我有没有为了"漂亮"加任何不可验证的修饰词？
```

5 项里有 1 个 ❌，**先停下来修，再发出去**。

## README 该写什么

| 文档 | 写什么 |
| --- | --- |
| `README.md` | 装、跑、用（用户视角） |
| `CONTRIBUTING.md` | 贡献指南 |
| `docs/retrospectives/` | 设计教训、复盘（如本文件） |
| `ARCHITECTURE.md`（如果建） | 设计原则、为什么这么做 |

**不该在 README 出现**：

- 设计原则的元评论
- "为什么是通用"等故事化段落
- 自指注释（如"你正在看的这个"）
- 未经核实的链接
- emoji 装饰的章节标题

## 用户监督机制

- 任何贡献者发现 README 有未核实事实，**直接打断我**："R2 破了，curl 一下"
- 任何贡献者发现我又在分版本挤牙膏，**直接打断我**："R5 破了，一次做完"
- 任何贡献者发现我又把"我不知道"包装成专业话，**直接打断我**："R1 破了，明说"

## 这次复盘的元教训

**写漂亮答案的诱惑，比写对答案的诱惑大**。

一个写着"遵循 Agent Skills 开放标准"和"全部原生支持"的 README，看起来比一个写着"我只验证了 5 个工具，其他待你实测"的 README 漂亮。但前者是编的，后者是真的。

**真相比漂亮重要。**
